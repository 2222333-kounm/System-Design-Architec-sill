# design-system-architect Skill 项目构建分析与改进建议

> 生成日期：2026-06-03
> 项目：sill — Apple 中国官网首页重构

---

## 一、项目构建过程回顾

### 1.1 项目概况

`design-system-architect` 是一个 Claude Code Skill，核心能力是从网页 URL 或设计截图**逆向工程**出完整的设计系统，输出结构化的 Design Token 体系和 `DESIGN.md` 规范文档。

本次任务流程：
```
用户提交 URL (apple.com.cn)
  → 使用 design-system-architect Skill
  → 分析网站提取设计元素（颜色/字体/间距/组件）
  → 生成 DESIGN.md 设计规范
  → 用户确认规范
  → writing-plans 生成 10 Task 实现计划
  → subagent-driven-development 逐 Task 执行（含审查循环）
  → 产出 index.html + css/style.css 完整页面
  → finishing-a-development-branch 收尾
```

### 1.2 使用到的 Skills 链

| Step | Skill | 作用 |
|------|-------|------|
| 1 | `brainstorming` | 探索需求，确认方案（HTML+CSS 重构） |
| 2 | `design-system-architect` | 分析网站，生成 DESIGN.md |
| 3 | `writing-plans` | 将设计规范转为 10 Task 实现计划 |
| 4 | `subagent-driven-development` | 逐 Task 执行 + 双重审查（spec + code quality） |
| 5 | `finishing-a-development-branch` | 工作收尾，提供合并/PR/保留选项 |

### 1.3 实际成效

| 指标 | 数据 |
|------|------|
| 总代码行数 | index.html: 342 行 + css/style.css: 809 行 = **1,151 行** |
| 设计规范 | DESIGN.md: 6 章完整规范，约 17KB |
| 总 Commits | **12 个**（含修复 commits） |
| 子 Agent 调用 | 10+次 implementer + 3 次 reviewer |
| 页面组件覆盖 | 导航栏 / 3x Hero / 6 张产品卡片 / CTA 栏 / 页脚 / 暗色模式 |
| 响应式断点 | 3 档（734px / 1068px / 1440px+） |

## 二、Skill 插件的优势分析

### 2.1 架构设计优点

1. **技能链完整** — 从需求探索 → 设计分析 → 计划生成 → 执行实现 → 收尾，覆盖了项目开发全生命周期
2. **模块化程度高** — 每个技能职责单一（brainstorming 只管需求、writing-plans 只管计划），组合使用很灵活
3. **质量门禁完善** — subagent-driven-development 的双重审查（spec compliance + code quality）有效防止低质量代码流入

### 2.2 设计分析准确性

从 Apple CSS 中提取的设计数据（颜色值 `#0071E3`、行高 `1.07143`、字间距 `-0.022em`、胶囊圆角 `980px`）准确度极高，与 Apple 官方设计完全一致。

### 2.3 代码生成效率

Agent 驱动的逐 Task 执行让 1,151 行代码在较短时间内完成生成，且代码结构清晰、注释完整。

## 三、Skill 插件的不足与改进建议

### 🔴 严重问题（影响可用性）

#### 1. **缺少真实的 CSS 解析引擎**

**现状：** 当前 analysis 全靠 WebFetch + grep 组合拳从 HTML/CSS 文件中手动提取颜色、字体等值。这种方式：
- 无法获取 computed styles（不渲染页面）
- 无法处理 CSS 变量链（如 `var(--sk-button-background)` 的最终值需要手动追踪）
- 对 JavaScript 渲染型网站完全无效（如 SPA、SSR 页面）
- 易受 Apple CDN 防盗链影响

**建议：** 集成一个浏览器引擎（Puppeteer/Playwright）或至少支持以下能力：
- 获取计算后的 CSS 自定义属性值
- 支持截图 + 多模态分析回退
- 对 SPA 页面先判断是否存在 `_next`、`react-root` 等特征，再决定解析策略

**优先级：** ⭐⭐⭐⭐⭐（最高 — 当前设计分析方式不可靠）

#### 2. **图片资源为外部 CDN 依赖**

**现状：** 生成的代码中所有产品图片链接直接引用 Apple CDN（`https://www.apple.com.cn/home/images/...`）：
- 面临防盗链（CORS/Referer）问题，可能在无浏览器环境无法加载
- 外部依赖不稳定，Apple 可随时更改路径

**建议：**
- 添加图片下载/本地化机制，生成静态页面时自动下载图片到本地 `images/` 目录
- 对于无法下载的图片，自动生成渐变/纯色占位图作为回退
- 在计划中明确标注外部图片资源的风险，提供替代方案

**优先级：** ⭐⭐⭐⭐

### 🟠 重要问题（影响开发体验）

#### 3. **design-system-architect 分析输出与 coding 之间断层**

**现状：** DESIGN.md 规范文档和实际代码之间存在信息脱节：
- DESIGN.md 定义了 `--color-primary-500: #0071E3`，但实现代码的 CSS 变量名可能不一致
- 按钮样式在规范中以表格形式存在，但实际 CSS 需要手写
- 没有「规范 → 代码」的自动转换机制

**建议：** 将 DESIGN.md 中的数据与代码生成打通：
- 从 DESIGN.md 中自动提取 CSS 变量定义，生成 `tokens.css`
- 支持 `design-system-architect` 输出 JSON 格式的 Machine-readable Token，供后续工具链消费
- 在技能中内建「Token → CSS/Sass/SCSS」转换器

**优先级：** ⭐⭐⭐⭐

#### 4. **Web 分析能力弱，遇反爬即失败**

**现状：** `curl` + `grep` 方式对现代网站几乎不可用：
- Apple.com.cn 的 CSS 文件内含大量 `.lang(xx)`、`@media` 条件编译代码
- 多个 CSS 文件之间有变量依赖（main.css 定义变量，home.css 消费）
- 无法区分全局样式和组件样式

**建议：**
- 引入 DOM 解析库（jsdom/cheerio）做结构化分析
- 对 CSS 做 AST 级别的解析，分离变量定义、组件样式、媒体查询
- 支持从 CSS 源码中识别 `:root` 变量块并自动提取

**优先级：** ⭐⭐⭐

#### 5. **没有用户自定义选项**

**现状：** 生成 DESIGN.md 到最终代码是固定流程，用户无法干预：
- 不能指定「想要浅色主题」「不用 SF Pro 字体」「使用国内 CDN」
- 不能选择响应式断点值（Apple 用 734/1068，但用户可能想要 768/1024）
- 不能选择「纯 HTML」还是「Tailwind CSS」输出风格

**建议：**
- 在 brainstorming 阶段增加风格选项表单
- 支持模板引擎：同一个 DESIGN.md 可输出多种技术栈（纯 CSS / Tailwind / Styled Components）
- 添加交互式配置步骤，让用户选择「我要深色为主」「按钮风格要多圆角/少圆角」

**优先级：** ⭐⭐⭐

### 🟡 可优化问题（提升质量）

#### 6. **CSS Design Token 变量命名不一致**

**现状：** DESIGN.md（规范）和 style.css（实现）的变量命名有差异：
```
DESIGN.md:  --font-family-cn
style.css:  --font-cn

DESIGN.md:  --color-button-dark-hover  # 有定义
style.css:  硬编码 #272729              # 没使用变量
```

**建议：**
- 在技能中内置命名一致性校验器
- Token 命名应以 DESIGN.md 为唯一数据源（Single Source of Truth）
- 在计划生成阶段就自动检查 Token 命名的一致性

**优先级：** ⭐⭐⭐

#### 7. **外部 CDN 图片路径硬编码**

**现状：** 所有产品图片引用了 Apple 官方的 CDN 路径：
```html
<img src="https://www.apple.com.cn/home/images/heroes/iphone-16-pro/iphone-16-pro-large.jpg">
```
如果 Apple 更改路径、添加版本号或更换图片，页面将显示 404。

**建议（扩展第 2 点）：**
- 实现一个「静态资源本地化」功能，下载图片到项目目录
- 使用 `<picture>` + 多分辨率 srcset
- 对缺失图片提供统一回退样式

**优先级：** ⭐⭐⭐

#### 8. **Agent 生成过程中 token 消耗较大**

**现状：** subagent-driven-development 模式虽然质量高，但每个 Task 调用 1 个 implementer + 2 个 reviewer，存在明显资源浪费：
- review 有时发现的只是微调问题（如 `transition: all` 改为 `transition: background`）
- 简单任务（如 Task 3 公告横幅只有 40 行 CSS）仍然需要完整的三 agent 流程

**建议：**
- 引入「任务复杂度评估」：简单任务（<50 行代码）可跳过 code quality review，只做 spec review
- implementer 在自我审查时自动做质量检查，减少 reviewer 的重复劳动
- 对纯 CSS 任务可合并执行（2-3 个相关 Task 一起提交）

**优先级：** ⭐⭐

#### 9. **暗色模式支持不完整**

**现状：** 当前只在亮色模式下做了主色定义，暗色模式覆盖了部分 CSS 变量：
- Hero 区的 `.tile-dark` 背景 `#000000` 在暗色模式下可以工作但不是最佳
- 部分 `@media (prefers-color-scheme: dark)` 中的变量覆盖不完整
- 缺少 `accent-color`、`color-scheme` 等 meta 标记

**建议：**
- 在 DESIGN.md 中强化暗色模式章节，定义完整的暗色变量体系
- 使用 `color-scheme: light dark` 声明
- 在 root 级别使用 `light-dark()` 函数（CSS Color Level 5）

**优先级：** ⭐⭐

#### 10. **缺少设计 Token 版本管理和增量更新机制**

**现状：** DESIGN.md 是静态文档，没有版本号管理机制：
- 第一次生成了 DESIGN.md v1.0.0，但用户修改后不知如何增量更新
- 没有 diff 机制来追踪设计变更
- 如果用户在项目中期修改了设计，需要重新分析

**建议：**
- 为 DESIGN.md 添加 Changelog 章节
- 设计增量更新流程：用户提供新截图/URL → 分析差异 → 生成增量更新
- 支持语义化版本号（MAJOR.MINOR.PATCH）追踪设计变更

**优先级：** ⭐⭐

## 五、未来方向展望

### 短期（可直接在 skill 内改进）

1. **提需求时增加 config 步骤**：在 brainstorming 阶段让用户选择输出风格（纯 CSS / Tailwind / 组件库）
2. **自动生成 tokens.css 文件**：从 DESIGN.md 中提取变量定义，独立为可导入的 CSS 文件
3. **增加 CSS 分析准确度**：在技能中加入更完善的 CSS 值提取规则

### 中期（需引入外部工具）

1. **集成 Puppeteer/Playwright**：真实渲染页面后取 computed styles
2. **图片下载 + 裁剪**：将 CDN 图片下载到项目 `assets/` 目录
3. **多模态分析扩展**：支持 Figma 设计文件导入（需接入 Figma API）

### 长期（平台级改进）

1. **Design Token 生态**：支持导出到 Design Token Community Group 标准格式（DTCG）
2. **跨项目 Token 复用**：建立设计 Token 库，项目间可共享主题
3. **CI/CD 集成**：在 CI 中自动检测设计偏离并报警

## 六、本次构建总结

`design-system-architect` Skill 在本次 Apple 官网重构中展现了强大的**设计逆向工程 → 规范文档化 → 代码生成**闭环能力。

其核心价值在于：
- **桥接设计与代码** — 将视觉设计转化为结构化的、AI 可读的设计规范
- **标准化输出** — 统一的 DESIGN.md 模板确保每次输出质量一致
- **技能链组合** — 与 brainstorming / writing-plans / subagent-driven-development 等技能配合，形成完整的开发工作流

主要瓶颈在于**前端分析层**：当前以字符串匹配为主的 CSS 提取方式在面对现代网站（CSS 变量链、JS 渲染、多 CSS 文件依赖）时准确性受限。这是最值得优先改进的方向。

> *"最好的设计是让人感受不到设计的存在。"* — Apple 设计哲学
> 最好的工具也是。当 Skill 能让从设计到代码的链路如行云流水般自然时，它才真正成熟。
